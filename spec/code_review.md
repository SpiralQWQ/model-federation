# 代码审查 — 管线阶段

> **所属管线**：Phase 2（执行）→ Phase 3（质检）→ Phase 4（交付）
> **入口条件**：用户说"审查/Review/CR/检查代码"
> **质量门禁**：5 个维度全部审查完成 + Qwen/GLM 质检 ≥9.5

---

## 步骤 1：正确性审查

- **检查点**：
  - 逻辑是否完整？边界条件是否处理？
  - 有无 off-by-one / 空指针 / 类型错误？
- **状态输出**：`[CR] 正确性: 通过/发现N个问题`

## 步骤 2：安全性审查

- **检查点**：
  - 有无硬编码密钥？SQL注入/路径遍历/XSS？
  - 输入校验是否充分？
- **状态输出**：`[CR] 安全性: 通过/发现N个问题`

## 步骤 3：性能审查

- **检查点**：
  - 有无不必要的循环/重复计算/大对象分配？
  - 算法复杂度是否可控？
- **状态输出**：`[CR] 性能: 通过/发现N个问题`

## 步骤 4：可读性审查

- **检查点**：
  - 命名是否清晰？注释是否必要且准确？
  - 复杂度是否可控？有无 magic number？
- **状态输出**：`[CR] 可读性: 通过/发现N个问题`

## 步骤 5：兼容性审查

- **检查点**：
  - 是否依赖特定环境/版本？
  - PowerShell 5.1 是否兼容？路径分隔符？
- **状态输出**：`[CR] 兼容性: 通过/发现N个问题`

## 步骤 6：提交前自检清单

```
[CR] 自检清单:
[CR] .gitignore 已包含 daily_data*.json: [是/否]
[CR] 无 C:\Users\ 硬编码路径: [是/否]
[CR] 无 API Key/Token 明文: [是/否]
[CR] CLAUDE.md 变更已走确认流程: [是/否]
```

- **规则**：全部为"是"方可提交
- **状态输出**：`[CR] 自检清单: 全部通过`

## 出口条件

- 输出物：审查报告，格式为：
  ```
  [严重] 行N: 问题描述（根因）
  [轻微] 行N: 问题描述
  [建议] 说明
  ```
- 签章：`[CR] 审查完成`

---

## 质检门禁（Phase 3 — Qwen + GLM 各自独立审查，均≥9.5 方通过）

### 检查项细则

| 检查项 | 具体判定标准 | Qwen 评分 | GLM 评分 |
|--------|------------|:---------:|:--------:|
| 5 维度全覆盖 | 正确性/安全性/性能/可读性/兼容性 5 个维度均输出 `[CR]` 状态。缺任一维度 → 扣 4 分 | /10 | /10 |
| 严重问题标注 | 发现的安全/正确性问题已明确指出行号和根因。只说"有问题"不指出位置 → 扣 3 分 | /10 | /10 |
| 自检清单全部通过 | `.gitignore` / 无硬编码路径 / 无明文密钥 / CLAUDE.md 确认流程 4 项均为"是"。任一为"否" → 扣 3 分 | /10 | /10 |
| 审查报告分级 | 输出必须按 `[严重] 行N: 问题描述` / `[轻微] 行N: 问题描述` / `[建议] 说明` 格式。混为一谈不分级 → 扣 2 分 | /10 | /10 |

### 通过条件

```
Qwen 评分 ≥ 9.5 AND GLM 评分 ≥ 9.5 → [质检] 裁决: 通过
Qwen 评分 < 9.5 OR GLM 评分 < 9.5 → [质检] 裁决: 返工（标注扣分项）
```
