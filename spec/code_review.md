# 代码审查 — Pre-submit 检查清单与标准

> 每次提交前的强制代码审查流程。防止"理论正确、实测翻车"的 38 次血泪教训重演。

## 提交前检查清单

提交前逐项确认，任何一项不通过则禁止提交：

- [ ] **语法检查**：Python 用 `python -m py_compile <file>` 或 `import` 验证无语法错误
- [ ] **submit.zip 验证**：解压后结构完整，入口文件存在且可 import
- [ ] **注释头更新**：文件 docstring 中的日期/分数/改动描述已更新为最新
- [ ] **单变量原则**：一次提交只改一个变量/一个逻辑点，禁止堆叠改动
- [ ] **备份覆盖**：修改前用备份文件硬覆盖（`infer_<日期>_<分数>_<改动>.py`），不用 git 回退
- [ ] **build_env.sh**：内容只能是 `echo "build env success"`，禁止包含 `pip install` 或其他网络依赖
- [ ] **精度检查**：`register_buffer` 不受 `half()` 影响，需单独处理
- [ ] **无盲狙操作**：Windows 无 CUDA GPU，每行张量操作必须有理论依据

## 代码审查标准

审查维度及通过门槛：

| 维度 | 标准 | 否决条件 |
|------|------|---------|
| 正确性 | 逻辑无歧义、边界条件覆盖 | 存在未处理的 shape 不匹配或除零 |
| 效率 | 无冗余计算、无 O(n^2) 可优化 | 存在可避免的显式循环 |
| 可读性 | 变量命名自解释、关键步骤有注释 | 混淆的魔法数字或无意义缩写 |
| 稳定性 | 异常路径有处理、数值范围有保护 | NaN/inf 传播路径未截断 |

## Qwen 预审查 + GLM 语义检查 工作流

1. **QwenMax 预审查**：代码修改后先发给 QwenMax，检查语法正确性与逻辑漏洞
2. **GLM 语义检查**：QwenMax 通过后，GLM 对修改语义进行一致性验证，输出 0-10 评分
3. **通过标准**：GLM 评分 >= 9.0 视为通过
4. **未通过**：回到 QwenMax 修改阶段；连续 3 轮未过则触发 DeepSeek 仲裁

> 百度优化目录下的代码修改，另需强制执行三方审核（详见 `collaboration.md`），此处的预审查流程是前置步骤。
